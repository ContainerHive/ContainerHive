# web/report

React app that renders the ContainerHive image report. It compiles to a **single self-contained HTML file** which is embedded into the Go binary and served/generated by `pkg/report/generator.go`.

## Dev workflow

```bash
yarn                 # first time only
yarn run dev         # local dev server with hot reload (uses mockData.ts)
yarn run build       # produces dist/index.html
make generate        # embeds dist/index.html into the Go binary (run from repo root)
```

After changing UI code: `yarn build` → `make generate` → rebuild the Go binary.

## Tech stack

- **Vite 6** + **React 19** + **TypeScript**
- `vite-plugin-singlefile` — bundles everything (JS, CSS, fonts) into a single `index.html`
- `vite-plugin-webfont-dl` — downloads Google Fonts locally so the output is fully offline

## Key files

| Path                             | Purpose                                                     |
|----------------------------------|-------------------------------------------------------------|
| `src/types.ts`                   | TypeScript interfaces matching the Go report model          |
| `src/mockData.ts`                | Sample data used by `npm run dev`                           |
| `src/main.tsx`                   | React entry point + React Router setup                      |
| `src/App.tsx`                    | Gallery view (searchable image grid)                        |
| `src/components/ImageDetail.tsx` | Detail view (tags, versions, build args, SBOM)              |
| `src/ThemeContext.tsx`           | Light/dark theme via React Context + localStorage           |
| `index.html`                     | HTML template — **all CSS lives here**, not in `.css` files |

## Data injection

At runtime the Go generator injects report JSON into `window.__REPORT_DATA__`. `src/main.tsx` reads it via `getReportData()`, falling back to `mockData.ts` when the variable is absent (i.e. during local dev).

Injection placeholders in `index.html`:
- `/*INJECT_JSON_DATA*/` — the full `ProjectReport` JSON
- `/*INJECT_GENERATED_AT*/`, `/*INJECT_SOURCE*/`, `/*INJECT_REGISTRY*/` — metadata strings

## Design system

See **`../../DESIGN.md`** for the full specification. Key rules to follow:

### Two themes
- **Light — "The Precision Architect":** white-gray canvas, navy `#006591` primary
- **Dark — "The Observability Monolith":** deep navy `#0b1326` base, cyan `#7bd0ff` primary, teal `#3cddc7` tertiary

Theme is set via a `data-theme` attribute on `<html>`. All CSS is written as variables scoped to `[data-theme="light"]` / `[data-theme="dark"]`.

### The No-Line rule
**No 1px solid borders for structural separation.** Use background color shifts between surface tiers instead. When a visible boundary is strictly required for accessibility, use `outline_variant` at 20% opacity (light) or 15% opacity (dark).

### Typography
- **Space Grotesk** — headings, repo/image names
- **Inter** — body text, metadata, SHA hashes

### CSS tokens
All design tokens are CSS variables defined in `index.html`. Use them — don't hardcode hex values. Examples: `var(--primary)`, `var(--surface-container-high)`, `var(--on-surface-variant)`.

### Styling notes
- All styles are inline in `index.html` — keep them there, do not create separate `.css` files
- Use `var(--...)` tokens consistently; check `index.html` for the full token list
- Cards/rows: no borders, separate with gap; hover = background shift to `surface_container_high`
- Inputs: no box border, bottom-only `outline_variant` at 20% opacity; focus = bottom border in `primary` (light) / `tertiary` (dark)
- Chips/tags: `border-radius: 9999px` (light) or `0.125rem` (dark)
